# Locust тестирование

## Краткое описание

Тестирование проводилось с помощью библиотеки `Locust`.


## 🔥 Запуск тестирования

### 1. Запуск генерации `short_url`

Запустите модуль генерации:

```shell
uv run locustfiles/short_link_creater.py
```

И собственно дождитесь когда сгенерируется 200 тысяч записей в базе данных. (Это сделано для того чтобы `PostgreSQL` не создавал кэш, например для 4 записей которые тысячу раз запрашиваются по кругу.) Это очень быстрая операция поскольку я это оптимизировал.

### 2. Запуск тест панели `Locust`

Запустите панель командой:
 
```shell
locust -f "locustfiles/locust.py" --host=http://127.0.0.1:8000
```

(Флаг --host показывает нашей библиотеке что за сервис мы будем тестировать, его url).

1. (`Nuber of users`)В первом поле надо будет выбрать пиковое число пользователей.

2. (`Ramp up`)Во втором поле надо будет выбрать кол-во юзеров появляющихся в секунду ДО пикового числа.
3. Нажмите старт и наслаждайтесь!

## Однопоточное тестирование

Все тесты проводились по 10 минут каждый!

### Тест 1

#### Как проводился

`Locust`, `10` пользователей пик, `1` польз/сек, host: локальный `Docker` контейнер `FastAPI` + `PostgreSQL`.

#### Результаты

(Всё в ms)

- **RPS:** `33`
- **Задержка:** 
	- Median < `5`.
	- p95 <= `5`.
	- p99 <= `4`.
- **Ошибки:** `0%`.

#### Технический вывод

Для приложения как фоновый шум. Нагрузки абсолютно никакой. Задержки минимальны. Ошибок нет.

### Тест 2

#### Как проводился

`Locust`, `100` пользователей пик, `10` польз/сек, host: локальный `Docker` контейнер `FastAPI` + `PostgreSQL`.

#### Результаты


(Всё в ms)

- **RPS:** `325`
- **Задержка:** 
	- Median <= `4`.
	- p95 <= `75`.
	- p99 <= `60`.
- **Ошибки:** `0%`.

#### Технический вывод

Почти тоже самое что и 1 тест. Маленькая нагрузка для процессора. Только вот возможно было изменение условий поскольку были запущены другие программы в фоне. Ошибок нет. Иногда проскакивают в 5 и в 1 процентах случаев задержки до 100 мс.

### Тест 3

#### Как проводился

`Locust`, `1000` пользователей пик, `50` польз/сек, host: локальный `Docker` контейнер `FastAPI` + `PostgreSQL`.

#### Результаты

(Всё в ms)

- **RPS:** `709`
- **Задержка:** 
	- Median <= `900`.
	- p95 <= `1500`.
	- p99 <= `1700`.
- **Ошибки:** `6%`.

#### Технический вывод

Тут уже приблизились к максимуму, не самого сервиса, а к максимуму моего железа(пк). Процессор зашел за грань в `90%` нагрузки. И поэтому посыпались ошибки и начались сильные задержки в ответе пользователю. Иногда проскакивало около `2000` мс ответа. Но `709` rps держит стабильно +-.
